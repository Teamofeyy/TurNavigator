# Экспериментальная оценка

## Цель

Документ фиксирует воспроизводимый MVP-набор метрик для оценки прототипа СППР TravelContext перед защитой.

## Команда запуска

```bash
cd /Users/teamofey/magistr-diploma/backend
UV_CACHE_DIR=../.uv-cache uv run python -m app.tools.evaluate_recommender --json
```

Скрипт:

- строит синтетические сценарии по нескольким городам;
- формирует persisted-профиль пользователя;
- генерирует рекомендации;
- собирает маршрут с учетом стартовой точки отеля, если она есть в каталоге;
- считает агрегированные метрики;
- сохраняет артефакты в [docs/results/recommender-evaluation.json](/Users/teamofey/magistr-diploma/docs/results/recommender-evaluation.json) и [docs/results/recommender-evaluation.csv](/Users/teamofey/magistr-diploma/docs/results/recommender-evaluation.csv).

## Метрики

### `precision@k`

Доля объектов в top-k, которые совпали с интересами пользователя (`matched_interests` не пустой).

### `coverage`

Доля категорий, покрытых в выдаче, относительно всего набора категорий каталога города.

### `budget-fit`

`1`, если маршрут укладывается в бюджет пользователя, и `0`, если нет.

### `time-fit`

`1`, если маршрут укладывается во временной лимит поездки, и `0`, если нет.

### `explanation usefulness`

Эвристика по тексту explanation:

- `1.0`, если explanation содержит и ссылку на интерес пользователя, и ссылку на ограничения;
- `0.5`, если есть только одно из двух;
- `0.0`, если explanation слишком общее.

### `context robustness`

Стабильность top рекомендаций после мягкого изменения контекста. В текущем скрипте увеличиваются бюджет и допустимая дистанция, после чего считается Jaccard overlap между двумя top-наборами.

## Интерпретация

- `precision@5=1.0` означает, что в текущем demo-наборе top выдачи полностью совпадает с синтетическими интересами.
- `coverage=0.286` показывает, что даже при идеальной релевантности система пока охватывает мало категорий, потому что каталог ограничен.
- `budget-fit` и `time-fit` на уровне `1.0` подтверждают, что текущие moderate/relaxed сценарии собираются без выхода за лимиты.
- `explanation_usefulness=1.0` подтверждает, что explanation уже отражает не только интересы, но и ограничения профиля.

## Ограничения текущей методики

- Метрики считаются на фиксированном demo-каталоге с очень малым числом кандидатов: в текущем прогоне это всего `2-3` candidate POI на сценарий.
- Поэтому итоговые метрики выглядят идеальными и не могут считаться полноценным подтверждением качества production-рекомендателя.
- `coverage` занижен не из-за алгоритма как такового, а из-за бедности импортированного набора категорий.
- `explanation usefulness` пока эвристическая и не основана на экспертной разметке.

## Что нужно сделать для усиления оценки

- расширить импорт хотя бы до `50-100` POI на город;
- зафиксировать один более богатый импортированный snapshot каталога для воспроизводимых запусков;
- добавить ручную экспертную оценку relevance top-5;
- сравнить несколько конфигураций весов `compromise_strategy` и `trust_level`;
- выделить отдельный сценарий для маршрутов с обязательным возвратом в отель.
