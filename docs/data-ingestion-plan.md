# Data Ingestion And Enrichment Plan

## 1. Зачем это нужно для диплома

Текущий MVP уже умеет:

- принимать пользовательский контекст;
- ранжировать POI по интересам, бюджету и ограничениям;
- строить маршрут;
- сохранять обратную связь.

Но для соответствия теме магистерской работы этого недостаточно. По ТЗ системе нужен не только рекомендатель, но и модуль получения, нормализации и обогащения данных о точках интереса. Именно поэтому в проект добавляется отдельный ETL-контур.

Связь с задачами работы:

- **5.2** — профиль пользователя должен сопоставляться не с ручными тегами, а с нормализованными признаками POI;
- **5.3** — персонализация рекомендаций должна опираться на реальные категории, описания, стоимость, фото и контекст;
- **5.4** — прототип должен включать модуль профиля, контекста, рекомендателя, объяснения и журнал решений, а также практический слой данных;
- **5.6** — требуется описать архитектуру, модель данных, риски и хранение информации.

## 2. Источники данных

### OpenTripMap

Используется как основной источник для популярных туристических объектов:

- достопримечательности;
- музеи;
- видовые точки;
- исторические объекты;
- часть описаний и preview-изображений.

### OpenStreetMap / Overpass

Используется как основной источник для контактных и прикладных данных:

- `website`;
- `phone`;
- `opening_hours`;
- `tourism`, `amenity`, `leisure`, `historic`, `shop`;
- инфраструктурные POI: кафе, рестораны, гостиницы, парки, магазины.

### Wikidata / Wikipedia / Wikimedia Commons

Используется как слой обогащения:

- `wikidata_id`;
- статья Wikipedia;
- краткое описание;
- thumbnail и media metadata;
- информация об авторе и лицензии изображения.

## 3. Каноническая модель POI

Внутри системы каждая точка интереса хранится в единой структуре:

- `name`;
- `category`;
- `subcategory`;
- `latitude`, `longitude`;
- `address`;
- `description`;
- `opening_hours`;
- `website`;
- `phone`;
- `wikidata_id`;
- `wikipedia_title`;
- `wikipedia_url`;
- `wikimedia_commons`;
- `osm_tags`;
- `interests`;
- `interest_source`;
- `primary_image`;
- `images`;
- `source_links`;
- `popularity_score`;
- `data_quality_score`;
- `data_freshness_days`;
- `last_enriched_at`.

## 4. Что делается автоматически

### 4.1. Автоматическая нормализация интересов

Для существующих seed-данных и будущих импортов система больше не полагается только на ручные интересы. В `backend/app/services/poi_enrichment.py` добавлена автоматическая инференция интересов по:

- `category`;
- `subcategory`;
- `osm_tags`;
- ключевым словам в названии;
- ключевым словам в описании.

В результате у каждого POI есть поле `interest_source`:

- `manual`;
- `automatic`;
- `manual+automatic`.

### 4.2. Автоматическая оценка полноты данных

Для каждого объекта вычисляется `data_quality_score`. Оценка учитывает наличие:

- описания;
- сайта;
- телефона;
- часов работы;
- `wikidata_id`;
- Wikipedia-ссылки;
- изображений;
- `source_links`;
- интересов;
- координат.

### 4.3. Автоматическая оценка популярности

Если у источника есть рейтинг, он используется как сигнал популярности. Если нет, `popularity_score` оценивается эвристически по косвенным признакам:

- наличие Wikidata;
- наличие Wikipedia;
- наличие изображений;
- наличие сайта;
- наличие телефона;
- наличие часов работы;
- тип объекта.

## 5. ETL-контур

В `backend/app/services/data_import.py` закладывается единый слой импорта.

Он отвечает за:

1. построение URL и запросов к OpenTripMap, Overpass, Wikidata и Wikipedia;
2. преобразование внешних payload в канонический POI;
3. обогащение POI описанием и изображениями;
4. дедупликацию и merge записей из нескольких источников;
5. подготовку данных к сохранению в локальную БД.

Основные функции:

- `build_opentripmap_places_url(...)`
- `build_opentripmap_detail_url(...)`
- `build_overpass_query(...)`
- `build_wikidata_entity_url(...)`
- `build_wikipedia_summary_url(...)`
- `normalize_opentripmap_place(...)`
- `normalize_overpass_element(...)`
- `enrich_with_wikimedia_context(...)`
- `deduplicate_and_merge_candidates(...)`
- `merge_poi_records(...)`

## 6. Стратегия хранения

### 6.1. Что хранить в PostgreSQL

#### `cities`

- базовые сведения о городе;
- координаты;
- population;
- внешние id.

#### `pois`

- каноническая запись POI;
- бизнес-поля для рекомендаций;
- описание;
- контактные данные;
- category/subcategory;
- текущие score и свежесть.

#### `poi_source_links`

- `poi_id`;
- `provider`;
- `external_id`;
- `url`;
- `license`;
- `last_synced_at`;
- при необходимости `raw_payload_jsonb`.

#### `poi_images`

- `poi_id`;
- `provider`;
- `original_url`;
- `thumbnail_url`;
- `source_page_url`;
- `license`;
- `author`;
- `attribution_text`;
- `width`, `height`;
- `is_primary`.

### 6.2. Как хранить изображения

Для MVP изображения лучше **не хранить бинарно в PostgreSQL**.

Нормальная стратегия:

- хранить URL изображений;
- хранить метаданные лицензии и атрибуции;
- при необходимости позже кешировать картинки в объектное хранилище:
  - S3;
  - Cloudflare R2;
  - Supabase Storage.

## 7. Очередность реализации

### Этап 1. Уже делается

- расширение POI schema под изображения и provenance;
- автоматическая нормализация `interests`;
- automatic quality/popularity enrichment;
- audit существующих seed-данных.

### Этап 2. Следующий

- перевод seed-набора на импортный pipeline;
- загрузка популярных POI по каждому из текущих MVP-городов;
- enrichment описаний, сайтов, телефонов и изображений.

Практическая цель для MVP-каталога:

- минимум `100` качественных POI на каждый из 6 текущих городов;
- quality threshold на первом проходе: `data_quality_score >= 0.55`;
- целевой pipeline:
  - Overpass даёт прикладные POI и контакты;
  - OpenTripMap даёт популярные туристические POI;
  - Wikidata / Wikipedia / Wikimedia дают описание и фото.

Идея набора:

1. первый проход без ключей — Overpass-only, чтобы быстро получить десятки дополнительных объектов;
2. второй проход с `OPENTRIPMAP_API_KEY` — добавить популярные и туристически значимые места;
3. третий проход — Wikimedia-enrichment, чтобы нарастить фото и описание;
4. после каждого импорта использовать `catalog_quality_report` и смотреть, сколько не хватает до 100.

### Этап 3. После этого

- переход на PostgreSQL;
- сохранение импортированных POI в БД;
- refresh strategy по свежести;
- расширение количества городов и объектов.

## 8. Практический результат для системы

После внедрения ETL-модуля система будет:

- подбирать рекомендации не по полностью ручным данным, а по автоматически нормализованным POI;
- иметь реальную базу фотографий, описаний и contact metadata;
- поддерживать масштабирование на большее число городов;
- лучше соответствовать магистерскому ТЗ как СППР, а не просто как UI-демо с seed-данными.
