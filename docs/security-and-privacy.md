# Security And Privacy

## Что обрабатывает система

TravelContext обрабатывает:

- интересы пользователя и его предпочтения по путешествию;
- бюджет, темп, ограничения по времени и доступности;
- выбранный город и стартовую точку маршрута;
- обратную связь по построенному маршруту;
- историю запросов, рекомендаций и маршрутов в decision log.

В MVP это уже относится к персонализированным пользовательским данным, даже если система пока не хранит ФИО и паспортные данные.

## Основные риски

- deanonymization через комбинацию города, отеля и редких предпочтений;
- утечка decision log, в котором есть снимки профиля и истории маршрутов;
- несанкционированный доступ к административной БД;
- повреждение данных при миграциях или невалидных snapshot payload;
- зависимость от внешних API и риск подмешивания некачественных или вредоносных данных.

## Минимальные меры защиты

### На backend

- ограничить доступ к PostgreSQL только из доверенной сети или через VPN;
- пометить `/profiles`, `PATCH /profiles/{profile_id}` и `/decision-logs` как demo/admin endpoints до внедрения auth;
- хранить `DATABASE_URL`, API-ключи и токены только в `.env`, а не в коде;
- включить отдельные роли БД для чтения и записи;
- логировать ошибки без вывода полного профиля пользователя в stderr;
- валидировать все входные payload через Pydantic-схемы.

### Для MVP-демо

Сейчас в коде нет пользовательской аутентификации, поэтому persisted профили и decision logs нельзя считать production-ready пользовательским контуром. Для защиты ВКР это допустимо, если явно зафиксировать ограничения:

- `/profiles` используются как demo-слой сохраненных профилей без личного кабинета;
- `/decision-logs` считаются исследовательским/admin endpoint для анализа качества СППР;
- следующий этап внедрения должен добавить хотя бы API key для demo-access и полноценный user/session auth для production.

### На уровне данных

- сокращать retention decision log до разумного окна для экспериментов;
- не хранить свободный адрес отеля, если выбран вариант из каталога POI, а сохранять ссылку на `poi_id` и координаты;
- отделить аналитические выборки от production decision log;
- описать политику удаления данных пользователя по запросу.

Минимальный план retention/delete для следующего этапа:

- `decision_logs` - ограничение хранения, например 90-180 дней для исследовательского контура;
- `profiles` и `trip_requests` - soft-delete или hard-delete по запросу пользователя;
- экспорт и очистка feedback перед публикацией учебных датасетов.

### На frontend

- не хранить чувствительные данные в localStorage без необходимости;
- не отправлять лишние поля в аналитические сервисы;
- показывать пользователю, что профиль и feedback сохраняются для улучшения системы.

## Соответствие защите ВКР

Для магистерской защиты достаточно показать:

- состав обрабатываемых персонализированных данных;
- угрозы и последствия их компрометации;
- технические и организационные контрмеры;
- ограничения MVP и план доведения до production-grade уровня.

## Что рекомендуется сделать следующим шагом

- добавить аутентификацию и разграничение ролей;
- ввести soft-delete и policy retention для `profiles`, `trip_requests`, `decision_logs`;
- вынести decision log snapshots в отдельную схему БД;
- добавить security review внешних ETL-источников;
- оформить отдельный раздел по требованиям к ПДн и правовым основаниям обработки.
