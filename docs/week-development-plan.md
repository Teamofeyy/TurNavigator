# План разработки до конца недели

Текущая дата планирования: 21 апреля 2026 года.

Цель недели: получить рабочий MVP веб-приложения, которое позволяет выбрать город, интересы, бюджет и ограничения, после чего показывает персонализированные рекомендации и простой маршрут с объяснениями.

Обязательный минимум:

- GitHub-репозиторий с кодом;
- frontend с основной страницей планирования;
- backend API на FastAPI;
- база данных или временная SQLite-база;
- стартовый набор городов;
- загрузка или seed-набор POI;
- алгоритм ранжирования рекомендаций;
- экран результата с объяснениями;

- загрузка POI из Overpass API;

- деплой до июня.

На этой неделе не нужно делать полный каталог всех городов России и всех заведений.

Для защиты достаточно показать архитектуру, которая масштабируется, и рабочий прототип на 3-10 городах.

Рекомендуемый MVP-набор:

- Москва;

- Санкт-Петербург;

## 4. План API

### 4.1 Health

`GET /health`

Назначение: проверить, что backend работает.

Ответ:

```json
{
  "status": "ok"
}
```

### 4.2 Города

`GET /cities`

Назначение: вернуть список доступных городов.

Ответ:

```json
[
  {
    "id": 1,
    "name": "Казань",
    "region": "Республика Татарстан",
    "latitude": 55.7961,
    "longitude": 49.1064,
    "pois_count": 24
  }
]
```

`GET /cities/{city_id}/pois`

Назначение: вернуть точки интереса по городу.

Параметры:

- `category`;
- `limit`;
- `offset`.

### 4.3 Профиль пользователя

`POST /profiles`

Назначение: сохранить профиль предпочтений пользователя.

Тело запроса:

```json
{
  "interests": ["culture", "food", "history"],
  "budget_level": "medium",
  "max_budget": 15000,
  "pace": "moderate",
  "max_walking_distance_km": 8,
  "preferred_transport": "walking",
  "explanation_level": "detailed"
}
```

### 4.4 Планирование

`POST /trip-requests`

Назначение: создать запрос на планирование.

Тело запроса:

```json
{
  "city_id": 1,
  "profile": {
    "interests": ["culture", "food", "history"],
    "budget_level": "medium",
    "max_budget": 15000,
    "pace": "moderate",
    "max_walking_distance_km": 8
  },
  "days_count": 2,
  "daily_time_limit_hours": 8
}
```

`POST /recommendations/generate`

Назначение: вернуть ранжированный список объектов.

Ответ:

```json
{
  "trip_request_id": 10,
  "recommendations": [
    {
      "poi_id": 101,
      "name": "Казанский кремль",
      "category": "history",
      "score": 0.92,
      "rank": 1,
      "explanation": "Объект выбран из-за высокого совпадения с интересами culture и history, высокой значимости и доступности по маршруту."
    }
  ]
}
```

`POST /routes/build`

Назначение: собрать маршрут из рекомендованных объектов.

Ответ:

```json
{
  "route_id": 3,
  "total_distance_km": 6.4,
  "total_time_minutes": 310,
  "estimated_budget": 4200,
  "points": [
    {
      "order": 1,
      "poi_id": 101,
      "name": "Казанский кремль",
      "latitude": 55.7996,
      "longitude": 49.1055
    }
  ],
  "explanation_summary": "Маршрут построен с умеренным темпом и не превышает заданный бюджет."
}
```

### 4.5 Обратная связь

`POST /feedback`

Назначение: сохранить оценку пользователя.

Тело запроса:

```json
{
  "route_id": 3,
  "rating": 5,
  "liked_poi_ids": [101, 104],
  "disliked_poi_ids": [108],
  "comment": "Маршрут удобный, но хотелось меньше музеев."
}
```

## 5. План backend

### 5.1 Первый рабочий backend

Сначала backend должен уметь:

- запускаться локально;
- отдавать `/health`;
- отдавать список городов;
- отдавать POI из seed-файла или базы;
- принимать запрос планирования;
- возвращать рекомендации.

На этом этапе база может быть SQLite, чтобы не тратить время на инфраструктуру.

### 5.2 Модели данных

Минимальные модели для первой версии:

- City;
- PointOfInterest;
- TripRequest;
- Recommendation;
- DecisionLog.

UserProfile можно сначала хранить прямо внутри TripRequest как JSON, а позже вынести в отдельную таблицу.

### 5.3 Сервисы backend

`data_import.py`

- читает seed-файлы;
- позже загружает POI из внешних API;
- нормализует категории.

`recommender.py`

- считает score;
- фильтрует объекты по бюджету и интересам;
- сортирует рекомендации.

`route_builder.py`

- на MVP сортирует точки по близости;
- считает примерное расстояние;
- позже подключает OSRM/OpenRouteService.

`explanations.py`

- формирует объяснения без LLM;
- позже может быть дополнен LLM-слоем.

## 6. План работы с данными

### 6.1 Самый быстрый путь

Для первой рабочей версии использовать seed-данные:

- `backend/app/seeds/cities.json`;
- `backend/app/seeds/pois.json`.

Это позволит быстро сделать интерфейс, API, рекомендации и скриншоты.

### 6.2 Потом добавить импорт

После рабочего MVP добавить импорт из одного источника:

Вариант A: OpenTripMap.

- проще получить туристические объекты;
- есть категории;
- удобно для достопримечательностей.

Вариант B: OpenStreetMap Overpass API.

- больше открытых данных;
- есть кафе, гостиницы, магазины;
- сложнее нормализовать.

Для этой недели лучше сначала seed, потом один импорт.

### 6.3 Нормализация категорий

Внутренние категории:

```text
culture
history
nature
food
accommodation
shopping
entertainment
service
transport
```

Пример маппинга:

```text
tourism=museum -> culture
tourism=attraction -> history
historic=* -> history
leisure=park -> nature
amenity=cafe -> food
amenity=restaurant -> food
tourism=hotel -> accommodation
shop=* -> shopping
```

## 7. План frontend

### 7.1 Главный экран

Должен содержать:

- выбор города;
- выбор интересов кнопками;
- бюджет;
- темп поездки;
- количество дней;
- кнопку построения маршрута.

### 7.2 Экран результата

Должен содержать:

- список рекомендованных объектов;
- объяснение по каждому объекту;
- примерный бюджет;
- примерное время;
- карту с точками;
- кнопку оценки результата.

### 7.3 Что можно упростить

Если карта не успевает:

- сначала показать список объектов и координаты;
- потом добавить Leaflet/MapLibre.

Если backend не успевает:

- frontend может временно работать с mock JSON;
- потом заменить на реальные API-запросы.
